## 1. Theming architecture

### 1.1 Cascade layer (dossier item 6)

The **entire** `styles.css` moves into `@layer morass`. Per the cascade
spec, unlayered author CSS beats layered CSS, so consumer stylesheets and
`--m-*` token overrides win regardless of stylesheet load order. The
import-order footgun dies structurally, not by documentation.

### 1.2 Full tokenization (dossier item 2, generalized)

Every color in `styles.css` becomes a `--m-*` token. New tokens:

| Token                    | Replaces                                                            | Default (light)                    |
| ------------------------ | ------------------------------------------------------------------- | ---------------------------------- |
| `--m-color-neutral`      | `.m-status-pill--neutral` color `#384642`                           | `#384642`                          |
| `--m-color-neutral-bg`   | `.m-status-pill--neutral` background `#eef1ef`                      | `#eef1ef`                          |
| `--m-color-sidebar-bg`   | `.m-app-frame__sidebar` background `#143330`                        | `#143330`                          |
| `--m-color-sidebar-text` | `.m-app-frame__sidebar` color `#f8faf8`                             | `#f8faf8`                          |
| `--m-color-step-bg`      | `.m-progress__step span` background `#e6e9e6`                       | `#e6e9e6`                          |
| `--m-color-input-bg`     | `.m-input` background `white`                                       | `#ffffff`                          |
| `--m-color-header-bg`    | `.m-app-frame__header` background `rgb(255 255 255 / 0.92)`         | `rgb(255 255 255 / 0.92)`          |
| `--m-color-backdrop`     | `.m-modal__backdrop` background `rgb(15 23 42 / 0.44)`              | `rgb(15 23 42 / 0.44)`             |
| `--m-color-on-primary`   | hard-coded `white`/`color: white` on primary/danger/active surfaces | `#ffffff`                          |
| `--m-focus-ring`         | `.m-input:focus` etc. outline color `rgb(29 118 111 / 0.25)`        | `rgb(29 118 111 / 0.25)`           |
| `--m-shadow-modal`       | `.m-modal__panel` box-shadow color                                  | `0 24px 80px rgb(15 23 42 / 0.24)` |

Existing `--m-color-*` tokens are unchanged. After this pass, the only raw
color values in `styles.css` live inside theme token blocks (`:root`,
`[data-m-theme="dark"]`, and the auto media block).

### 1.3 Theme contract (ratified: attribute themes + auto OS default)

A theme is a token set. Morass ships two; anyone can add more by writing a
token block. `light-dark()` was considered and rejected: it hard-wires
exactly two schemes, prevents subtree theming, and drags in `color-scheme`
side effects.

```css
@layer morass {
  :root {
    /* default (light) tokens */
  }
  [data-m-theme="dark"] {
    /* dark token overrides */
  }
  @media (prefers-color-scheme: dark) {
    :root:not([data-m-theme]) {
      /* dark token overrides, duplicated verbatim */
    }
  }
}
```

Contract, as documented in the README:

- **Default:** no attribute → light, except the OS prefers dark, in which
  case the auto block applies dark.
- **Force a theme:** set `data-m-theme="dark"` or `data-m-theme="light"` on
  `<html>` (or any subtree — theming a region works because tokens
  cascade). `"light"` matches no override block and disables the auto
  media block, so it pins the light defaults; this is the escape hatch for
  consumers not ready for surprise-dark.
- **Custom themes:** override `--m-*` tokens in your own (unlayered) CSS;
  they win over every morass block by layer rules. A named custom theme is
  just `[data-m-theme="yourname"] { … }` in consumer CSS.

The dark block appears twice in source (attribute selector + media query).
A unit test asserts the two blocks contain identical declarations so they
cannot drift (see §3).

### 1.4 Dark theme values

Dark values are chosen at implementation time against these requirements,
using the repo demo/tests for verification:

- Text-on-background pairs (`text`/`bg`, `text-muted`/`surface`,
  each tone `color`/`*-bg` pair) meet WCAG AA (4.5:1).
- Tone semantics keep their hue identity (danger reads red, warning amber,
  success green, info blue) on dark surfaces.
- Starting palette (tunable during implementation): bg `#121817`,
  surface `#1a2321`, surface-muted `#15201d`, text `#e6ebe9`, text-muted
  `#9fb0ac`, border `#2c3835`, primary `#3ba69d`, primary-strong
  `#5fc0b7`, on-primary `#0c1514`, tone text lightened / tone backgrounds
  as dark translucent tints (e.g. danger `#f97066` on `rgb(180 35 24 /
0.16)`), sidebar keeps `#143330` family, input-bg `#151d1b`, header-bg
  `rgb(26 35 33 / 0.92)`, backdrop `rgb(0 0 0 / 0.6)`, focus ring
  `rgb(95 192 183 / 0.35)`, step-bg `#243230`.

## 2. Component changes

### 2.1 AppFrame single-column fix (dossier item 1)

`.m-app-frame` defaults to `grid-template-columns: minmax(0, 1fr)`. When
the `sidebar` prop is present the component adds `m-app-frame--sidebar`,
which restores `248px minmax(0, 1fr)`. No API change; the ≤820px media
query keeps collapsing the sidebar variant to one column.

### 2.2 ButtonLink (dossier item 3)

New component rendering `<a>` with the existing `m-button` classes.

```ts
export interface ButtonLinkProps extends AnchorHTMLAttributes<HTMLAnchorElement> {
  icon?: ReactNode;
  variant?: ButtonVariant; // default "secondary", same as Button
}
```

Markup mirrors `Button` (icon span + label span). `.m-button` CSS gains
`text-decoration: none` so anchors render identically. Chosen over a
polymorphic `as` prop on `Button`: two small, honestly-typed components
beat one clever one at this library's size.

### 2.3 Card footer slot (dossier item 5)

`CardProps` gains `footer?: ReactNode`, rendered after the body:

```tsx
{
  footer ? <footer className="m-card__footer">{footer}</footer> : null;
}
```

`.m-card__footer`: top border `--m-color-border`, padding matching the
header. The homepage's consumer-styled `.go` link moves here at adoption
time.

### 2.4 Hero (dossier item 4)

```ts
export interface HeroProps extends HTMLAttributes<HTMLElement> {
  actions?: ReactNode; // CTA row, typically ButtonLink(s)
  eyebrow?: ReactNode;
  lede?: ReactNode;
  title: ReactNode; // ReactNode so inline emphasis spans work
}
```

Renders `<section class="m-hero">` with eyebrow (`m-eyebrow`, reused),
`<h1 class="m-hero__title">`, `<p class="m-hero__lede">`, and an optional
`m-hero__actions` row. Styling: large title scale, muted lede, generous
block padding — extracted from the homepage's `.hero`/`.lede` CSS but
token-driven.

### 2.5 PageSection (dossier item 4)

```ts
export interface PageSectionProps extends HTMLAttributes<HTMLElement> {
  label?: ReactNode; // section eyebrow, e.g. "What we make"
}
```

Renders `<section class="m-page-section">` with optional
`<p class="m-eyebrow">` label and children. `id` and other section
attributes pass through for anchor navigation. Divider rules between
sections stay consumer-side — they are page design, not component
structure.

## 3. Testing

Extend `src/components.test.tsx` in the existing style:

- **AppFrame:** with sidebar → has `m-app-frame--sidebar` and renders
  aside; without → no modifier class, no aside.
- **ButtonLink:** renders `<a href>` with `m-button m-button--<variant>`
  classes, icon slot, no `type` attribute.
- **Card:** footer renders in `m-card__footer`; absent when prop omitted.
- **Hero / PageSection:** structural render + attribute passthrough
  (`id`, `className` merge).

New `src/styles.test.ts` (reads `src/styles.css` as text):

- **Layer guard:** all rules live inside `@layer morass`.
- **Token completeness:** no raw color values (hex, `rgb(`, named colors)
  outside the three theme token blocks.
- **Theme sync:** the `[data-m-theme="dark"]` block and the
  `prefers-color-scheme` auto block declare identical property/value sets.

## 6. Risks

- **Surprise-dark on OS dark mode** for consumers with no theme attribute
  and no token remap. Accepted deliberately (Kevin, 07-15) in exchange for
  zero-wiring OS support; mitigated by the `data-m-theme="light"` pin and
  a CHANGELOG callout. Hub is unaffected (full unlayered token remap wins
  over all morass theme blocks).
- **`@layer` demotes morass specificity.** Consumer CSS that previously
  _lost_ to morass by specificity will now win. Reviewed as acceptable:
  consumers targeting `m-*` classes do so to override, and both known
  consumers' overrides are intentional.
- **Dark token block duplication** could drift; locked by the theme-sync
  test.
