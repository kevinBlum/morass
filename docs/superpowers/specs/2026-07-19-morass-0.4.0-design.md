# Morass 0.4.0 — Design

**Date:** 2026-07-19
**Source:** dossier #39 (machine-enforced contrast contract), direction and
scope ratified by Kevin in the 07-15 COO session; API shape, token
source-of-truth, and pair scope ratified 07-19.
**Positioning:** morass's differentiator is design contracts that are
machine-enforced, not documented. `@layer` guarantees override order;
`styles.test.ts` enforces no-raw-colors and theme sync. 0.4.0 extends the
contract to color _relationships_: a theme that violates contrast fails a
test instead of shipping.

## Goals

1. Export the built-in themes as data (`themes.light` / `themes.dark`),
   sync-enforced against the CSS token blocks.
2. Export `REQUIRED_PAIRS` (the contrast contract as data) and
   `validateTheme` (a pure WCAG checker over that contract), runnable in
   any consumer's CI.
3. Validate our own themes with it; fix palette values that fail rather
   than grandfathering them.
4. A11y credibility: Modal focus trap + Escape; Tabs arrow-key navigation.

## Non-goals

- No new components beyond the a11y work.
- No docs site.
- No runtime enforcement — `validateTheme` is a build/CI tool; it is never
  imported by component code.
- No non-text (3:1 border/focus-ring) contrast enforcement in v1. The
  built-in soft borders (`#d9ded8` on white, ~1.2:1) are a deliberate
  divider aesthetic; bringing them into contract would force a palette
  change on every consumer. Revisit in a later minor.
- No codegen: both the CSS token blocks and the TS theme objects are
  hand-written; tests make drift unshippable ("generic over clever",
  ratified 0.3.0).

## 1. `themes` module

`src/themes.ts` exports:

```ts
export type MorassTokenName = `--m-${string}`;
export type MorassTheme = Record<MorassTokenName, string>;
export const themes: { light: MorassTheme; dark: MorassTheme };
```

Values are hand-written and must mirror `styles.css` exactly:
`themes.light` ≡ the `:root` token block, `themes.dark` ≡ the
`[data-m-theme="dark"]` block. All tokens are included (colors, focus
ring, shadows) so the object is a faithful theme snapshot; `validateTheme`
only reads the tokens the contract names.

**Sync enforcement:** a new `styles.test.ts` case parses each CSS token
block (reusing `blockOf`/`declarationsOf`) into a record and asserts deep
equality with the TS objects — same pattern as the existing dark/auto sync
test. The existing dark/auto test remains; together they enforce
CSS-light ≡ TS-light and CSS-dark ≡ TS-dark ≡ CSS-auto.

Exported from the package root (`export * from "./themes"`).

## 2. Contrast contract (`src/contract.ts`)

### 2.1 `REQUIRED_PAIRS`

A plain exported data array; each entry:

```ts
interface RequiredPair {
  fg: MorassTokenName; // text token
  bg: MorassTokenName[]; // compositing chain, topmost first,
  // ending in an opaque base
  context: string; // where this renders, e.g.
  // "StatusPill danger text on pill bg over Card"
}
```

The chain exists because several dark-theme backgrounds carry alpha
(`--m-color-danger-bg: rgb(180 35 24 / 0.16)`): the checker composites
each layer over the next until opaque, then measures fg against the
result. Chains that start opaque are length 1.

The pair list is **usage-audited**: every fg/bg combination the components
actually render as text, including state variants. The audit is part of
implementation, but the known population is:

- `text` over each of: `bg`, `surface`, `surface-muted`, `input-bg`,
  `header-bg` (chain: over `bg`), `step-bg` (chain: over `surface`).
- `text-muted` over: `bg`, `surface`, `surface-muted`.
- `on-primary` over `primary` **and** over `primary-strong` (hover).
- Each tone `X` over its `X-bg` with the pill's real underlay chain
  (`surface`, then `bg`): danger, warning, success, info, neutral.
- `sidebar-text` over `sidebar-bg`.
- `primary` over `bg` and `surface` where components render it as
  standalone text (links / ghost variants), confirmed by audit.

**Threshold:** uniform **4.5:1** for every pair (WCAG AA normal text).
No large-text 3:1 carve-out — uniformity is simpler to reason about and
buys headroom.

### 2.2 `validateTheme`

```ts
function validateTheme(theme: MorassTheme): ValidationResult;

interface ValidationResult {
  ok: boolean;
  failures: PairFailure[];
}

interface PairFailure {
  fg: MorassTokenName;
  bg: MorassTokenName[];
  ratio: number; // computed, 2dp
  required: number; // 4.5
  context: string;
}
```

Pure and zero-dependency. Color parsing supports the forms the themes
actually use: `#rgb`/`#rrggbb`(/`#rrggbbaa`), `rgb(r g b)` and
`rgb(r g b / a)` (comma syntax accepted too). Compositing is standard
source-over; ratio is WCAG 2.x relative luminance.

**Loud failure:** a token named by the contract that is missing from the
input, or a value that doesn't parse, **throws** (with the token name) —
a typo must not pass as vacuously valid. Tokens outside the contract are
ignored, so consumers can pass a superset (e.g. everything scraped from
`getComputedStyle`).

Exported from the package root. Unit tests in `src/contract.test.ts`:
known-ratio fixtures (black/white 21:1, mid-grays), alpha-compositing
cases checked against hand-computed values, throw cases, and a regression
fixture reproducing the 0.2.1 bug (`#ffffff` on `#5bb89c` ≈ 2.4:1 must
fail).

### 2.3 Self-enforcement

New test: `validateTheme(themes.light).ok` and
`validateTheme(themes.dark).ok` are both `true`, with failures printed on
failure for diagnosis. If the initial audit finds current values failing
(muted text is the likely suspect), the palette values are adjusted in
the same PR — the validator drives; nothing is grandfathered. Any such
change is listed in the release notes.

## 3. Modal a11y

- **Escape** closes (keydown listener while open).
- **Focus trap:** Tab/Shift-Tab cycle within the panel's focusable
  elements; implemented as a small effect + keydown handler in
  `components.tsx`, no new dependencies.
- **Initial focus:** the panel container (`tabIndex={-1}`) on open —
  predictable regardless of content.
- **Focus return:** the element focused before open regains focus on
  close/unmount.

Existing behavior (backdrop click closes, `role="dialog"`,
`aria-modal`) is unchanged. Tests: testing-library keyboard interaction
— Escape fires `onClose`, Tab from the last focusable wraps to the
first, focus restored after close.

## 4. Tabs a11y

WAI-ARIA tablist pattern with automatic activation:

- Roving tabindex: active tab `tabIndex={0}`, others `-1`.
- **ArrowLeft/ArrowRight** move selection (wrapping), **Home/End** jump
  to first/last; moving selection focuses the newly active tab and fires
  `onValueChange`.

Tests: arrow/Home/End behavior, wrapping, tabindex attribution.

## 5. Docs

README gains a "Theme contract" section: the pair-contract story, a
consumer-CI example (the ratified API-shape snippet), and the
live-extraction recipe:

```js
const styles = getComputedStyle(document.documentElement);
const theme = Object.fromEntries(
  TOKEN_NAMES.map((n) => [n, styles.getPropertyValue(n).trim()]),
);
```

(`TOKEN_NAMES` derivable from `Object.keys(themes.light)`.)

## 6. Downstream (out of scope here)

After release: hub frontend adds a CI check calling `validateTheme` on
its remapped theme (replacing the one-off Playwright probe as the
first-line guard; the browser probe recipe stays documented for
composited end-to-end evidence). Webbery same, when it does its dark
pass.

## Testing summary

- `styles.test.ts`: existing invariants + CSS↔TS theme sync (both
  themes).
- `contract.test.ts`: parser/compositor/ratio fixtures, throw cases,
  0.2.1 regression, self-validation of both built-in themes.
- `components.test.tsx`: Modal Escape/trap/restore, Tabs keyboard
  pattern.

## Release

Version `0.4.0`, released through the proven OIDC dual-publish pipeline.
Behavior changes: none for mouse users; keyboard behavior of Modal/Tabs
improves; palette values change only if self-validation requires it
(release-noted).
